# Authentication & HTTP Context

When working on any source file in this project, be aware of the authentication system:

## Key Files

- **Token storage**: `src/utils/TokenManager.js` — singleton, stores `{ token, timestamp }` in `localStorage` under key `esp32_auth_token`
- **HTTP client**: `src/utils/HttpClient.js` — singleton, auto-injects `Authorization: Bearer <token>` on protected endpoints, dispatches `auth-error` events on 401/403
- **HTTP agent**: `src/utils/HttpRequestAgent.js` — singleton, higher-level wrapper with `login()`, `logout()`, `validateToken()`, `get()`, `post()` methods
- **Auth state**: `src/features/Auth/AuthSlice.js` — Redux slice with `loginUser`, `logoutUser`, `initializeAuth` thunks
- **Route guard**: `src/features/Auth/components/AuthGuard.jsx` — wraps entire app, validates token on startup
- **Middleware**: `src/middleware/authMiddleware.js` — handles `auth-error` events and 24h token expiration
- **API config**: `src/config/apiConfig.js` — all endpoints, public endpoint list, error messages
- **Types**: `src/types/auth.js` — JSDoc types: `AuthState`, `UserInfo`, `LoginCredentials`, `TokenData`

## Patterns to Follow

- Use `HttpRequestAgent` for API calls (not `HttpClient` directly) in services and thunks
- Use `skipAuth: true` option for public endpoints (login, health, status, wifi)
- Always handle `AbortError` separately when using abort signals
- Clear tokens on any auth failure to maintain clean state
- Use `ErrorHandlingService` for user-facing error messages
- Redux thunks should use `rejectWithValue()` for error propagation
